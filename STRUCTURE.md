## ① 화면 구조 — 총 5개

### A. 생성 도구 (내 PC / `tool/`) — 3개 화면

| # | 화면 | 핵심 내용 | 주요 액션 |
|---|------|----------|----------|
| 1 | **입력** | 경쟁사·키워드 입력, 프리셋 저장/불러오기, 수집 범위(24h·10개) 조정 | [브리핑 생성] |
| 2 | **생성 진행** | 소스별 수집 상태(네이버/웹), 요약 진행률, 부분 실패 표시 | (자동 진행) |
| 3 | **미리보기·편집** | 아카이브와 똑같은 스타일로 결과 표시, 항목 편집/삭제/순서, 제목·요약 수정 | [배포] |

> 2번은 1↔3 사이 상태 화면이라 원하면 1번에 통합 가능. 최소 구성은 **입력 → 미리보기·편집** 2개.

### B. 공개 아카이브 (GitHub Pages / `docs/`) — 2개 화면

| # | 화면 | 파일 | 핵심 내용 |
|---|------|------|----------|
| 4 | **카드 인덱스** | `index.html` | `data/index.json` 읽어 카드 그리드(날짜·제목·요약), 최신순 |
| 5 | **브리핑 상세** | `briefing.html?date=YYYY-MM-DD` | 해당 날짜 JSON 읽어 항목별 제목·요약·원문 링크 |

## ② 데이터 — 로컬 보관 vs 배포 스냅샷

**핵심 원칙: 배포되는 건 정제된 스냅샷뿐, 원재료와 키는 로컬에만.** 세 층으로 나눈다.

### 🔒 로컬 전용 (절대 배포 안 됨 · `.gitignore` 또는 `tool/` 내부)
- API 키 — 네이버 Client ID/Secret, 웹 검색 키, Claude 키 (`.env`)
- 검색 API 원본 응답(raw JSON), 수집 로그, 실패 기록
- 프리셋(자주 쓰는 경쟁사·키워드 세트)
- 편집 중인 임시 초안(배포 전)

### 📤 배포 스냅샷 — 상세용 `data/briefings/<date>.json`
```jsonc
{
  "id": "2026-07-14",
  "date": "2026-07-14",
  "title": "...",              // 브리핑 제목
  "summary": "...",            // 카드용 2~3문장
  "competitors": ["A","B"],
  "keywords": ["요금제","채용"],
  "items": [
    {
      "competitor": "A",
      "keyword": "요금제",
      "headline": "...",            // 항목 제목
      "summary": "...",             // 항목 요약 (Claude 생성)
      "source_url": "https://...",  // 원문 링크
      "published_at": "..."
    }
  ],
  "generated_at": "..."
}
```
**빠지는 것**: API 키, 원본 API 응답, 검색어·호출 횟수, 내부 점수/디버그 필드.
공개돼도 문제없는 "요약 + 원문 링크"만 남긴다.

### 📇 배포 스냅샷 — 카드용 `data/index.json` (경량, 카드만 그리는 용도)
```jsonc
[
  { "id":"2026-07-14", "date":"2026-07-14", "title":"...", "summary":"..." }
]
```
상세(`items`)는 뺀 최소 필드 → 인덱스 화면이 무거워지지 않게 분리.

> **원칙**: `items[].summary`는 원문 전체가 아니라 **요약**만. 저작권·민감정보 회피 + 스냅샷 경량화. 원문은 링크로만.

## ③ 사용자 사용 순서 (매일 아침)

```
1. tool/ 실행 (localhost 접속)
2. 입력 화면 → 프리셋 불러오기 or 경쟁사·키워드 입력
3. [브리핑 생성] → 네이버+웹 수집 → 중복 제거 → Claude 요약
4. 미리보기에서 확인 → 필요시 항목 편집/삭제, 요약 다듬기
5. [배포] → JSON 저장 + index.json 갱신 + git commit & push
6. GitHub Pages 자동 반영 → 공개 아카이브에 오늘 카드 추가됨
─────────────────────────────────────────
방문자: 아카이브 URL 접속 → 카드 목록 → 카드 클릭 → 상세 열람
```

## ④ 만드는 우선순위

**데이터 계약을 먼저 고정 → 결과물(아카이브)을 눈에 보이게 → 재료(도구)를 붙이는 순서.**
각 단계를 독립적으로 검증할 수 있다.

| 순서 | 무엇 | 왜 먼저 | 검증 방법 |
|:---:|------|--------|----------|
| **1** | **데이터 스키마 확정** + 손으로 쓴 더미 JSON 1~2개 | 두 부분의 유일한 계약. 여기가 흔들리면 전부 흔들림 | JSON이 ②와 일치 |
| **2** | **아카이브(`docs/`)** — 카드 인덱스 + 상세, 더미 JSON으로 | 결과 모양을 눈으로 확정. 도구 없이도 완성 가능 | 로컬에서 열어 카드·상세 동작 |
| **3** | **GitHub Pages 배포** (저장소 생성 + `gh` 로그인) | 공개 파이프라인을 일찍 뚫어둠 | 공개 URL에서 더미 카드 보임 |
| **4** | **생성 도구 — 수집** (네이버+웹, 중복 제거) | 입력→후보 항목까지 | 콘솔에 수집 결과 출력 |
| **5** | **생성 도구 — 요약**(Claude) + 미리보기·편집 | 후보→완성 브리핑 JSON | 미리보기가 아카이브와 동일 |
| **6** | **[배포] 연결** (JSON 저장 + git push) | 도구↔아카이브 E2E 완성 | 버튼 한 번에 실제 카드 추가 |
| **7** | 프리셋·필터·에러 처리 등 Should/Could | 핵심 흐름 뒤 개선 | — |

**첫 착수점은 1 → 2 → 3.**
도구(수집·요약·API 키) 없이도 **공개 아카이브가 실제로 배포되어 카드가 보이는 상태**까지 갈 수 있어, 가장 위험한 배포 파이프라인을 초반에 확인한다. `gh` 로그인도 이 3단계에서 처음 필요해진다.
