# PRD — 경쟁사 동향 브리핑 도구 & 아카이브

> **한 줄 요약**: 경쟁사 이름과 키워드를 입력하면, 검색 API로 그날의 동향을 자동 수집하고 Claude API로 요약해 "오늘 챙겨볼 브리핑"을 보여주는 로컬 웹 도구. 브리핑을 '배포'하면 GitHub Pages에 카드 아카이브로 쌓인다.

- **작성일**: 2026-07-14
- **작성자**: juckwoo@gmail.com
- **핵심 결정**:
  - 데이터 수집 = 검색 API 자동 수집 (**네이버 + 범용 웹 검색 하이브리드**)
  - 요약 = **Claude API 자동 요약**
  - 저장소 = **모노레포(한 저장소)**, 아카이브는 GitHub Pages 공개
  - 아카이브 구현 = **순수 HTML + JS (빌드 없음)**
  - 수집 기본값 = **최근 24시간 / (경쟁사 × 키워드)당 최대 10개**

## 0. 시스템 개요

두 개의 독립된 부분으로 구성된다. 경계는 **"배포(publish)"** 액션이다.

```
┌─────────────────────────────┐        publish        ┌──────────────────────────────┐
│  Part A. 브리핑 생성 도구     │  ── briefing.json ──▶ │  Part B. 브리핑 아카이브       │
│  (내 PC / localhost)         │   (git commit+push)   │  (GitHub Pages / 공개)         │
│                             │                       │                              │
│  입력 → 수집 → 요약 → 미리보기 │                       │  카드 인덱스 + 개별 상세 페이지 │
└─────────────────────────────┘                       └──────────────────────────────┘
        비공개 · 내 PC에서만 실행                              공개 · 정적 사이트
```

- **Part A**는 내 PC에서만 도는 비공개 도구. API 키를 다루고, 자동 수집·요약·편집·미리보기를 담당한다.
- **Part B**는 GitHub에 배포되는 공개 정적 사이트. **로직이 없고** 데이터(브리핑 JSON)만 읽어 카드로 렌더링한다.
- 둘의 유일한 접점은 Part A가 생성해 커밋하는 **브리핑 데이터 파일**이다. 이 계약(스키마)만 지키면 두 부분을 독립적으로 개발·교체할 수 있다.

# Part A. 브리핑 생성 도구 (로컬)

## A-1. 목적
매일 아침, 최소한의 입력으로 "오늘 경쟁사에 대해 챙겨봐야 할 것"을 정리된 브리핑으로 받아본다.

## A-2. 사용자 흐름
1. 로컬에서 도구 실행 (`localhost`)
2. **경쟁사 이름**(복수)과 **키워드**(복수) 입력 — 프리셋 저장/불러오기 지원
3. **[브리핑 생성]** 클릭
   - 검색 API로 각 (경쟁사 × 키워드)에 대한 최신 항목을 수집 (기본: 최근 24~48시간)
   - 중복 제거·정렬 후 Claude API로 항목별 핵심 요약 + 전체 제목 생성
4. **미리보기** 화면에서 결과 확인 — 항목 편집/삭제, 제목·요약 수정 가능
5. **[배포]** 클릭 → 브리핑을 아카이브 데이터로 저장하고 GitHub에 push

## A-3. 기능 요구사항

| ID | 요구사항 | 우선순위 |
|----|---------|:-------:|
| A-F1 | 경쟁사·키워드 입력 폼 (복수 입력, 프리셋 저장/불러오기) | Must |
| A-F2 | 검색 API로 경쟁사×키워드 최신 항목 자동 수집 (기간·개수 설정) | Must |
| A-F3 | 수집 항목 중복 제거 및 관련도/시간순 정렬 | Must |
| A-F4 | Claude API로 항목별 핵심 요약 + 브리핑 제목 자동 생성 | Must |
| A-F5 | 생성된 브리핑 미리보기 (카드 형태로 아카이브와 동일 스타일) | Must |
| A-F6 | 항목 수동 편집·삭제·순서 조정, 요약/제목 수정 | Should |
| A-F7 | **배포**: 브리핑을 JSON으로 저장 → git add/commit/push | Must |
| A-F8 | API 키를 로컬 환경변수/`.env`로 안전하게 관리 (커밋 금지) | Must |
| A-F9 | 수집·요약 실패 시 부분 성공 처리 및 에러 표시 | Should |
| A-F10 | 하루 1회 이상 재생성 시 같은 날짜 브리핑 덮어쓰기/버전 처리 | Should |

## A-4. 기술 구성(확정)
- **런타임**: Node.js. 로컬 서버 + 브라우저 UI (예: Vite + 간단한 프런트, 또는 Next.js 로컬 실행)
- **검색 API (하이브리드)**:
  - **네이버 검색 API** — 국내 뉴스/블로그 커버리지 담당
  - **범용 웹 검색 API** (Brave/Serper 등) — 해외·영문 자료 커버리지 담당
  - 두 소스 결과를 합친 뒤 **URL/제목 기준 중복 제거**, 시간순 정렬 (A-F3)
- **요약**: Claude API (`claude-sonnet-5` 등) — 원문 → 제목·핵심 요약 생성
- **수집 기본값**: 최근 **24시간**, (경쟁사 × 키워드)당 최대 **10개** (설정에서 조정 가능)
- **출력**: 아래 A-5 스키마의 JSON 파일을 `data/` 경로에 기록 후 push (모노레포 내부)
- **비밀정보**: `.env`에 API 키(네이버 Client ID/Secret, 웹 검색 키, Claude 키), `.gitignore`로 커밋 차단

## A-5. 출력 데이터 계약 (Part A ↔ Part B 인터페이스)
배포 시 브리핑 하나당 아래 형태의 JSON을 생성한다. **이 스키마가 두 부분의 유일한 계약이다.**

```json
{
  "id": "2026-07-14",
  "date": "2026-07-14",
  "title": "경쟁사 A, 신규 요금제 발표 — 오늘의 핵심 3건",
  "summary": "카드에 노출될 2~3문장 핵심 요약",
  "competitors": ["경쟁사 A", "경쟁사 B"],
  "keywords": ["요금제", "채용", "투자"],
  "items": [
    {
      "competitor": "경쟁사 A",
      "keyword": "요금제",
      "headline": "항목 제목",
      "summary": "항목별 핵심 요약",
      "source_url": "https://...",
      "published_at": "2026-07-14T08:12:00Z"
    }
  ],
  "generated_at": "2026-07-14T09:00:00Z"
}
```

카드 목록용으로는 `id/date/title/summary`만 있는 경량 인덱스(`index.json`)를 함께 갱신한다.

# Part B. 브리핑 아카이브 (GitHub 공개 페이지)

## B-1. 목적
배포된 브리핑들이 카드 목록으로 쌓이는 공개 아카이브. 방문자는 날짜별 브리핑을 훑고, 눌러서 상세를 본다.

## B-2. 화면 구성
1. **인덱스(카드 목록) 페이지**
   - 브리핑마다 카드 1장: **날짜 · 제목 · 핵심 요약** 표시
   - 최신순 정렬, 날짜/키워드 필터(선택), 반응형 그리드
   - 카드 클릭 → 해당 브리핑 상세로 이동
2. **개별 상세 페이지**
   - 제목, 날짜, 대상 경쟁사·키워드
   - 항목 리스트: 항목 제목 · 요약 · 원문 링크
   - 인덱스로 돌아가기

## B-3. 기능 요구사항

| ID | 요구사항 | 우선순위 |
|----|---------|:-------:|
| B-F1 | `index.json`을 읽어 카드 목록 렌더링 (날짜·제목·요약) | Must |
| B-F2 | 카드 최신순 정렬 및 반응형 그리드 레이아웃 | Must |
| B-F3 | 카드 클릭 시 개별 브리핑 상세 페이지 열림 | Must |
| B-F4 | 상세 페이지: 항목별 제목·요약·원문 링크 표시 | Must |
| B-F5 | GitHub Pages로 자동 배포 (push 시 공개) | Must |
| B-F6 | 날짜/키워드 필터·검색 | Could |
| B-F7 | 데이터 없을 때 빈 상태(empty state) 안내 | Should |

## B-4. 기술 구성(확정)
- **형태**: **순수 HTML + JS** 정적 사이트 (빌드 없음). 서버·DB 없음
- **호스팅**: **GitHub Pages** (저장소 push → 자동 공개)
- **데이터**: Part A가 커밋한 `index.json` + `briefings/<date>.json`을 클라이언트에서 `fetch`해 렌더링
- **렌더링**: `index.html`이 `data/index.json`을 읽어 카드 그리드 생성 / `briefing.html?date=YYYY-MM-DD`가 해당 JSON을 읽어 상세 렌더링
- **모노레포 구조(확정)**:
  ```
  repo/
    tool/                 # Part A. 브리핑 생성 도구 (로컬 실행, 비공개 로직)
      ...                 # Node 서버 + UI, .env(gitignore)
    docs/                 # Part B. 아카이브 (GitHub Pages 공개 루트)
      index.html          # 카드 목록
      briefing.html       # 상세 (?date=YYYY-MM-DD)
      assets/             # css/js
      data/
        index.json        # 카드용 경량 인덱스
        briefings/
          2026-07-14.json
    .gitignore            # .env, 로컬 산출물 제외
    README.md
  ```
  > GitHub Pages는 `docs/` 폴더를 게시 루트로 지정 → 공개되는 건 아카이브뿐, `tool/` 코드는 노출돼도 비밀정보는 `.env`에만 있어 안전.

## 공통: 배포(Publish) 동작 정의
"배포" 버튼을 누르면 Part A가 수행하는 일:
1. 확정된 브리핑을 `data/briefings/<date>.json`으로 저장
2. `data/index.json`에 카드 항목(날짜·제목·요약) 추가/갱신
3. `git add → commit("brief: <date>") → push`
4. GitHub Pages가 자동 반영 → 공개 URL에서 카드 확인

> 이 흐름을 쓰려면 `gh` 로그인 또는 git 원격 인증이 필요하다 (이미 `gh` 설치됨, 로그인만 남음).

## 비기능 요구사항
- **비용**: 검색 API·Claude API 호출 비용이 발생. 하루 1회 생성 기준으로 소액 예상 — 호출량 상한 설정
- **보안**: API 키는 절대 공개 저장소에 커밋 금지. Part B에는 어떤 비밀정보도 포함하지 않음
- **공개 범위**: 아카이브는 공개 페이지 → 원문 저작권/민감정보 유의 (요약+원문 링크 중심, 전문 복제 지양)
- **성능**: 아카이브는 정적 파일이므로 빠름. 브리핑 수가 많아지면 인덱스 페이지네이션 고려
